HUST第三节实验阶梯税问题引深

一切的起因是C语言程序设计课第三次实验的一道程序设计题

题目如下：

程序设计

假设工资税金按以下方法计算：x ＜ 1000元，不收取税金；1000 ≤ x ＜ 2000，收取5%的税金；2000 ≤ x ＜ 3000，收取10%的税金；3000 ≤ x ＜ 4000，收取15%的税金；4000 ≤ x ＜ 5000，收取20%的税金；5000 ≤ x，收取25%的税金。（注意税金的计算按照阶梯计税法，比如，工资为4500，那么税金=1000×5%* + *1000×10% + 1000×15% + 501×20%）。编写一个程序,输入工资金额，输出应收取税金额度，要求分别用if语句和switch语句来实现。

if与switch代码十分简单，不再赘述

神R提出了一个问题：能否不用分支语句实现？

结合以前提到过的闰年天数题，我想到了数组，用数组储存阶梯税率，对工资x进行处理后分段乘上税率求和即为答案。

但神R有较为优化的解，很快他给了出来

忽略掉一些奇奇怪怪的坑，显然这是我们目前为止能力范围内较优解。

接下来的问题是，这种写法比if或者switch好在哪

“我可以将这个数组和最大档存在配置文件里, 运行的时候现场读”​

“我也可以在运行的时候提供修改功能, 即改即生效, 都不用把服务停下来”​

“分支结构应该仅用在针对bool变量判断的功能分支上”​

“功能分流应该用更加解耦的router或者mapper结构”​

“数据计算就更不应该if switch了，将数据跟逻辑完全解耦开，代码质量才能上去”​

以我的能力，现在理解这些还有些难度。我目前领会到的就是，在大型项目中，这种代码写法显然更容易维护和更改功能。做不好动态管理，一些比较重要的二维码崩了就得维护很长时间，这期间如果软件完全停运，将给经济社会人民种种方面带来诸多不便。

可能这就是理想主义者的悲哀所在。

pps.算是留个坑，日后有了更深的理解回来把它填上
